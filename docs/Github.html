1. GitHub là gì?

Git

Git là công cụ quản lý phiên bản. Nó ghi lại lịch sử thay đổi của mã nguồn để bạn có thể quay lại phiên bản cũ, so sánh thay đổi và làm việc song song.

GitHub

GitHub là nền tảng lưu trữ repository trên cloud, hỗ trợ làm việc nhóm qua Issues, Pull Requests, Projects, Actions và Wiki.

Hiểu nhanh: Git giống như “máy ảnh chụp lịch sử code”, còn GitHub là “ngôi nhà chung” nơi cả nhóm cùng xem, góp ý, kiểm tra và triển khai code.
🌱

Lệnh hay dùng

git clone <url> git checkout -b feature/login git status git add . git commit -m "Add login page" git push origin feature/login

Nguyên tắc nhỏ mà xịn

  • Commit nhỏ, rõ mục đích.
  • Tên branch dễ hiểu: feature/cart, fix/login-bug.
  • Không commit file nhạy cảm: token, password, secret key.
  • Luôn pull/rebase trước khi tạo Pull Request nếu nhánh đã cũ.

3. Làm việc nhóm trên GitHub

Issues

Dùng để ghi nhận task, bug, ý tưởng hoặc yêu cầu thay đổi.

Nên có: mô tả, ảnh minh họa, bước tái hiện, tiêu chí hoàn thành.

Pull Requests

Nơi thảo luận code trước khi gộp vào nhánh chính.

Nên có: mô tả thay đổi, link issue, ảnh demo, checklist test.

Code Review

Giúp bắt lỗi sớm, chia sẻ kiến thức và giữ chất lượng code.

Nên review: logic, bảo mật, hiệu năng, readability, test.

Luồng làm việc nhóm đề xuất

1. Nhận task: chọn Issue trên board, đọc yêu cầu và hỏi lại nếu chưa rõ.
2. Tạo branch: branch riêng cho từng task, ví dụ feature/payment-ui.
3. Code & commit: commit theo từng phần có nghĩa, tránh commit quá lớn.
4. Mở Pull Request: mô tả thay đổi, gắn reviewer, chạy CI.
5. Review & sửa: trao đổi lịch sự, sửa theo góp ý, cập nhật PR.
6. Merge: chỉ merge khi đã pass test, được approve và không conflict.
🏃

4. GitHub theo chuẩn Agile

Agile là cách làm sản phẩm theo vòng lặp ngắn, nhận phản hồi thường xuyên và cải tiến liên tục. GitHub hỗ trợ Agile rất tốt thông qua Issues, Labels, Milestones và Projects.

Ánh xạ Agile vào GitHub

Agile Trên GitHub
Product Backlog Danh sách Issues/User Stories
Sprint Milestone hoặc Project iteration
Task Board GitHub Projects: Todo, In Progress, Review, Done
Definition of Done Checklist trong Issue/PR + CI pass + review approve

Mẫu User Story

Là một người dùng,

tôi muốn đăng nhập bằng email,

để truy cập tài khoản cá nhân.

Acceptance Criteria:

  • Nhập email/password hợp lệ thì đăng nhập thành công.
  • Sai thông tin thì hiển thị lỗi rõ ràng.
  • Có test cho trường hợp thành công và thất bại.

5. CI/CD với GitHub Actions

CI là gì?

Continuous Integration là tự động build, lint, test mỗi khi có code mới. Mục tiêu: phát hiện lỗi sớm trước khi merge.

CD là gì?

Continuous Delivery/Deployment là tự động chuẩn bị hoặc triển khai ứng dụng lên môi trường staging/production sau khi code đạt chuẩn.

Pipeline CI/CD trực quan

flowchart TD
  A["Developer push code"] --> B["GitHub Actions trigger"]
  B --> C["Install dependencies"]
  C --> D["Lint & Format check"]
  D --> E["Run tests"]
  E --> F["Build app"]
  F --> G{"Tất cả pass?"}
  G -- "Không" --> H["❌ Báo lỗi trong PR"]
  G -- "Có" --> I["✅ Cho phép merge"]
  I --> J["Deploy staging/production"]

Ví dụ workflow GitHub Actions cho Node.js

name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Run lint
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build

6. Quy ước chất lượng nên áp dụng

✓
Branch protection: bảo vệ nhánh main, không cho push trực tiếp.
✓
Required reviews: ít nhất 1–2 người approve trước khi merge.
✓
Required status checks: CI phải pass trước khi merge.
✓
Conventional Commits: dùng tiền tố như feat:, fix:, docs:, refactor:.
✓
PR template: chuẩn hóa mô tả thay đổi, cách test, ảnh demo, checklist.
✓
Secrets management: lưu token trong GitHub Secrets, không đưa vào code.

8. Checklist học nhanh

Người mới nên luyện

  1. Tạo repo cá nhân.
  2. Clone repo về máy.
  3. Tạo branch mới.
  4. Commit và push thay đổi.
  5. Mở Pull Request.
  6. Giải quyết conflict đơn giản.

Team nên thống nhất

  1. Quy tắc đặt tên branch.
  2. Quy tắc commit message.
  3. Definition of Done.
  4. Checklist review code.
  5. Quy trình CI/CD.
  6. Ai có quyền merge/deploy.

Tóm tắt siêu ngắn

Dùng GitHub hiệu quả nghĩa là: chia việc rõ bằng Issues, làm code trên branch riêng, review qua Pull Request, quản lý sprint bằng Projects/Milestones, và tự động kiểm thử - triển khai bằng GitHub Actions.

Khi nhóm làm đúng quy trình, code an toàn hơn, giao tiếp rõ hơn và sản phẩm ra phiên bản mới nhanh hơn.

🎉 Học GitHub vui vẻ — commit nhỏ, review kỹ, deploy tự tin!
