01

Mô hình Async thực chiến

FastAPI cho bạn hai cách viết handler — def và async def — và chọn sai gần như luôn là nguyên nhân số một khiến API "đột nhiên" chậm hẳn khi có tải thật.

Vòng lặp sự kiện: chỉ có một luồng duy nhất

Khi bạn khai báo async def, FastAPI chạy hàm đó trực tiếp trên event loop — một vòng lặp duy nhất xử lý hàng nghìn request cùng lúc bằng cách xen kẽ giữa các tác vụ đang chờ I/O (đợi database, đợi một API bên ngoài trả lời...). Cơ chế này chỉ hoạt động nếu mọi tác vụ trong hàm đều biết "nhường lượt" bằng từ khoá await.

Vấn đề nảy sinh khi bạn gọi một hàm chặn (blocking) — ví dụ thư viện requests, hoặc một vòng lặp tính toán nặng — bên trong một async def. Hàm đó không nhường lượt được, nên toàn bộ event loop bị đóng băng trong lúc nó chạy. Hậu quả: mọi request khác, kể cả endpoint /health chẳng liên quan gì, cũng phải xếp hàng chờ.

Hình 1 — Một lệnh gọi chặn bên trong async def làm nghẽn toàn bộ event loop, không chỉ request đang chạy.

Quy tắc chọn def hay async def

Tình huốngDùngVì sao
Gọi DB/API bằng thư viện hỗ trợ async (asyncpg, httpx.AsyncClient) async def + await Nhường event loop trong lúc chờ, không tốn thread
Gọi thư viện chỉ hỗ trợ đồng bộ (requests, driver DB cũ) def thường FastAPI tự đưa vào threadpool, không chặn loop chính
Tính toán CPU nặng (resize ảnh, parse PDF, suy luận mô hình trên CPU) def hoặc đẩy ra worker queue CPU nặng làm nghẽn cả threadpool lẫn event loop nếu để trong async def
main.py — minh hoạ đúng/sai
import httpx
import requests
from fastapi import FastAPI

app = FastAPI()

# ❌ SAI: async def nhưng gọi thư viện chặn bên trong
@app.get("/gia-vang-sai")
async def gia_vang_sai():
    r = requests.get("https://api.gia-vang.example/now")  # chặn cả loop!
    return r.json()

# ✅ ĐÚNG: dùng client bất đồng bộ, await đúng chỗ
@app.get("/gia-vang-dung")
async def gia_vang_dung():
    async with httpx.AsyncClient(timeout=3.0) as client:
        r = await client.get("https://api.gia-vang.example/now")
    return r.json()

# ✅ CŨNG ĐÚNG: thư viện chặn nhưng dùng def → FastAPI tự đưa vào threadpool
@app.get("/gia-vang-threadpool")
def gia_vang_threadpool():
    r = requests.get("https://api.gia-vang.example/now")
    return r.json()
Cách phát hiện sớm

Thêm một coroutine "canary" chạy nền: ngủ 100ms rồi đo thời gian thực tế thức dậy. Nếu độ trễ thực tế lệch xa hơn nhiều so với 100ms, event loop đang bị một tác vụ nào đó chiếm dụng. Đưa số liệu này vào dashboard như một chỉ số riêng — đừng chỉ nhìn CPU, vì loop có thể nghẽn ngay cả khi CPU rảnh (đang chờ một I/O chặn).

httpx.AsyncClient: dùng chung, không tạo mới mỗi request

Một lỗi phổ biến khác: tạo httpx.AsyncClient() mới bên trong mỗi handler. Mỗi client mới phải bắt tay TCP + TLS lại từ đầu, tốn 1-3 round-trip mạng cho mỗi request thay vì tái sử dụng kết nối đã mở (keep-alive). Giải pháp là tạo client một lần duy nhất trong lifespan, lưu vào app.state, rồi tái sử dụng cho mọi request.

main.py — client dùng chung qua lifespan
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Khởi tạo MỘT LẦN khi service start
    app.state.http = httpx.AsyncClient(
        timeout=httpx.Timeout(connect=2.0, read=5.0, write=5.0, pool=2.0),
        limits=httpx.Limits(max_connections=100, max_keepalive_connections=20),
    )
    yield
    # Đóng gọn gàng khi service shutdown
    await app.state.http.aclose()

app = FastAPI(lifespan=lifespan)

Tác vụ nền: BackgroundTasks vs hàng đợi thật (Celery/RQ)

BackgroundTasks của FastAPI chạy sau khi response đã gửi đi, nhưng vẫn trong cùng process. Nó phù hợp cho việc "được mất cũng không sao" — gửi log audit, làm ấm cache. Nhưng nếu pod bị restart giữa chừng (deploy mới, OOM), tác vụ đó biến mất không dấu vết — không retry, không log lỗi.

Với việc gì bắt buộc phải xảy ra — gửi email xác nhận đơn hàng, xử lý thanh toán, sinh báo cáo dài — cần một hàng đợi thật sự đứng sau bởi Redis/RabbitMQ (Celery, RQ, arq...). API chỉ nhận yêu cầu, đẩy vào hàng đợi, trả về 202 Accepted kèm mã job, rồi để worker xử lý độc lập.

Hình 2 — Mô hình job bất đồng bộ: nhận nhanh, xử lý đáng tin cậy, báo trạng thái qua polling hoặc webhook.
Quy tắc ghi nhớ

Nếu mất tác vụ đó khi pod restart là một lỗi nghiêm trọng đối với nghiệp vụ, nó không thuộc về BackgroundTasks.

Pydantic làm hàng rào dữ liệu

Schema không chỉ để validate đầu vào — nó còn là tuyến phòng thủ cuối cùng ngăn dữ liệu nhạy cảm rò rỉ ra ngoài.

Ba schema, không phải một

Sai lầm kinh điển: dùng một model cho cả nhận dữ liệu vào, lưu trữ, và trả dữ liệu ra. Khi model lưu trữ có trường is_admin, và client có thể gửi field đó trong body POST, bạn vừa mở ra lỗ hổng "mass assignment" — client tự phong mình làm admin. Ngược lại, nếu trả thẳng object ORM ra ngoài mà không lọc, bạn vô tình trả luôn hashed_password.

Giải pháp kỷ luật: tách ba model riêng biệt cho mỗi resource.

Hình 3 — Ba schema độc lập: input bị giới hạn, lưu trữ giàu thông tin, output là whitelist tường minh.
schemas/user.py
from pydantic import BaseModel, ConfigDict, EmailStr, Field, SecretStr

class UserCreate(BaseModel):
    """Những gì client ĐƯỢC PHÉP gửi lên khi đăng ký."""
    model_config = ConfigDict(extra="forbid")  # field lạ → 422, không bị bỏ qua âm thầm
    email: EmailStr
    password: SecretStr = Field(min_length=12)
    display_name: str | None = None

class UserPublic(BaseModel):
    """Những gì API trả về. Không bao giờ có password hay cờ nội bộ."""
    model_config = ConfigDict(from_attributes=True)  # đọc trực tiếp từ object ORM
    id: int
    email: EmailStr
    display_name: str | None

@app.post("/users", response_model=UserPublic, status_code=201)
def create_user(payload: UserCreate, db: Session):
    user = user_service.create(db, payload)
    return user  # dù object có 15 field, response_model chỉ cho 3 field đi qua
Đừng tin vào kỷ luật gọi hàm

Đừng dựa vào việc "nhớ" gọi .dict(exclude={"password"}) ở mọi nơi trả response — chỉ cần một chỗ quên là rò rỉ dữ liệu. Hãy biến việc lọc field thành cấu trúc (response_model riêng) thay vì quy ước (nhớ loại trừ field mỗi lần).

Phân biệt "không gửi" và "gửi null"

Với endpoint PATCH, JSON không tự phân biệt được giữa "client không gửi field này" (giữ nguyên) và "client gửi field này = null" (xoá giá trị). Pydantic v2 theo dõi field nào thực sự được client cung cấp qua model_dump(exclude_unset=True).

schemas/task.py — PATCH đúng cách
class TaskUpdate(BaseModel):
    """Mọi field đều optional, không có default mang ý nghĩa."""
    title: str | None = None
    note: str | None = None

@app.patch("/tasks/{task_id}")
def update_task(task_id: int, payload: TaskUpdate, db: Session):
    # exclude_unset=True: chỉ lấy field client THỰC SỰ gửi lên
    changes = payload.model_dump(exclude_unset=True)
    # {"note": None}  → khách muốn XOÁ note
    # {}               → khách không động đến field nào cả
    return repo.update(db, task_id, changes)

from pydantic import BaseModel, model_validator
from datetime import date

class BookingCreate(BaseModel):
    check_in: date
    check_out: date

    @model_validator(mode="after")
    def checkout_sau_checkin(self) -> "BookingCreate":
        if self.check_out <= self.check_in:
            raise ValueError("check_out phải sau check_in")
        return self

Database & Connection Pool

API thì stateless, nhưng giá trị thực luôn nằm ở dữ liệu — và đa số sự cố production đến từ hai chỗ: truy vấn N+1 và pool kết nối hết chỗ.

Session theo từng request

Mỗi request nên có session DB riêng, mở ra khi request bắt đầu, đóng lại khi kết thúc — kể cả khi có lỗi. Trong FastAPI, cách làm chuẩn là một dependency kiểu yield: phần trước yield chạy lúc setup, phần sau chạy lúc teardown, kể cả khi handler ném exception.

db/session.py
from sqlalchemy.orm import sessionmaker, Session
from sqlalchemy import create_engine

engine = create_engine(
    settings.database_url,
    pool_size=10,        # số kết nối "thường trực"
    max_overflow=5,      # cho phép vượt tạm khi cao điểm
    pool_pre_ping=True,   # kiểm tra kết nối còn sống trước khi dùng
    pool_recycle=1800,   # tái tạo kết nối cũ hơn 30 phút
)
SessionLocal = sessionmaker(bind=engine)

def get_db():
    db = SessionLocal()
    try:
        yield db
        db.commit()       # thành công → lưu thay đổi
    except Exception:
        db.rollback()      # lỗi → huỷ mọi thay đổi trong transaction
        raise
    finally:
        db.close()          # LUÔN trả kết nối lại pool
Vì sao finally quan trọng

Thiếu khối finally: db.close(), một nhánh lỗi không được xử lý sẽ làm rò rỉ kết nối — nó không bao giờ trả lại pool. Dưới tải cao và tỉ lệ lỗi tăng, pool cạn kiệt dần và toàn bộ service đứng hình, dù phần lớn request vẫn hợp lệ.

Vấn đề N+1 — kẻ thù số một của hiệu năng ORM

Khi bạn lấy 100 task rồi với mỗi task lại truy cập task.owner, ORM "lazy loading" mặc định sẽ bắn ra một query SELECT riêng cho mỗi owner. Kết quả: 1 query lấy task + 100 query lấy owner = 101 round-trip database, thay vì 1 hoặc 2.

Hình 4 — selectinload gom các truy vấn con thành một câu lệnh duy nhất bằng WHERE id IN (...).
repositories/task_repo.py — diệt N+1
from sqlalchemy import select
from sqlalchemy.orm import selectinload

# ❌ Lazy loading mặc định → N+1 khi vòng lặp truy cập task.owner
stmt_cham = select(Task).limit(100)

# ✅ selectinload: gom owner thành 1 truy vấn IN (...) duy nhất
stmt_nhanh = (
    select(Task)
    .options(selectinload(Task.owner))
    .limit(100)
)
# Quy tắc chọn nhanh:
#   selectinload  → quan hệ "nhiều" (to-many), tránh nhân bản dòng
#   joinedload    → quan hệ "một" (to-one), gộp 1 lần JOIN duy nhất

Tính toán kích thước connection pool

Theo định luật Little: số kết nối cần thiết ≈ số request/giây × thời gian giữ kết nối trung bình. Với 200 request/giây và mỗi truy vấn giữ kết nối 25ms, bạn chỉ cần khoảng 5 kết nối — ít hơn nhiều so với trực giác. Nhưng nếu nhân lên với số worker, con số có thể vượt giới hạn max_connections của Postgres rất nhanh.

Bảo mật: Auth & OWASP

Phần lớn sự cố bảo mật API thật sự không phải lỗi xác thực (authn) — mà là lỗi phân quyền (authz): người dùng đã đăng nhập nhưng đọc được dữ liệu của người khác.

Authentication vs Authorization

Authentication (authn) trả lời câu hỏi "bạn là ai" — mã 401 nghĩa là "tôi không biết bạn là ai, hãy đăng nhập lại". Authorization (authz) trả lời "bạn được phép làm gì" — mã 403 nghĩa là "tôi biết bạn là ai, nhưng không, bạn không được phép". Tách hai khâu này ra hai dependency riêng biệt giúp logic rõ ràng và dễ kiểm thử hơn.

deps.py — chuỗi dependency xác thực
from typing import Annotated
import jwt
from fastapi import Depends, HTTPException
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

def get_current_user(
    token: Annotated[str, Depends(oauth2_scheme)],
    db: Session = Depends(get_db),
) -> User:
    try:
        payload = jwt.decode(
            token, settings.jwt_secret,
            algorithms=["HS256"],   # GHIM thuật toán — không bao giờ tin header token
            issuer="api.vidu.com",
        )
    except jwt.InvalidTokenError:
        raise HTTPException(401, "Token không hợp lệ hoặc hết hạn")
    user = db.get(User, int(payload["sub"]))
    if user is None or not user.is_active:
        raise HTTPException(401, "Tài khoản không tồn tại hoặc bị khoá")
    return user

class RequireRole:
    """Dependency có tham số: kiểm tra quyền theo vai trò."""
    def __init__(self, *roles: str):
        self.roles = set(roles)

    def __call__(self, user: User = Depends(get_current_user)) -> User:
        if user.role not in self.roles:
            raise HTTPException(403, "Không đủ quyền")
        return user

admin_router = APIRouter(dependencies=[Depends(RequireRole("admin"))])

Vì sao phải ghim thuật toán JWT

Header của JWT khai báo thuật toán ký — và header này do kẻ tấn công kiểm soát hoàn toàn. Một số thư viện cũ tin tưởng mù quáng giá trị này: kẻ tấn công đặt "alg": "none" và gửi token không hề được ký. Vì vậy luôn truyền tường minh algorithms=["HS256"] khi giải mã — không bao giờ để token tự chọn cách xác thực chính nó.

BOLA — lỗ hổng đứng đầu danh sách OWASP API

Broken Object-Level Authorization (BOLA) xảy ra khi endpoint kiểm tra "bạn đã đăng nhập chưa" nhưng quên kiểm tra "đây có phải tài nguyên của bạn không". Ví dụ kinh điển: GET /orders/123 chỉ cần token hợp lệ, không kiểm tra order.owner_id == user.id — kẻ tấn công chỉ cần đổi số ID tuần tự để đọc dữ liệu của bất kỳ ai.

Hình 5 — BOLA xảy ra khi việc kiểm tra "có phải của bạn" tách rời khỏi câu truy vấn.
repositories/order_repo.py — chống BOLA tại tầng repository
class OrderRepository:
    def __init__(self, db: Session):
        self.db = db

    def get_owned(self, order_id: int, owner_id: int) -> Order | None:
        """Không có hàm get(id) trần trụi nào cho resource thuộc về user."""
        stmt = select(Order).where(
            Order.id == order_id,
            Order.owner_id == owner_id,   # lọc ngay tại SQL, không lọc sau khi load
        )
        return self.db.scalar(stmt)

@router.get("/orders/{order_id}")
def get_order(order_id: int, user: User = Depends(get_current_user), db: Session = Depends(get_db)):
    order = OrderRepository(db).get_owned(order_id, user.id)
    if order is None:
        raise HTTPException(404, "Không tìm thấy đơn hàng")  # 404, KHÔNG phải 403
    return order

Mật khẩu: chậm là một tính năng

Không bao giờ băm mật khẩu bằng SHA-256 — thuật toán này được thiết kế để nhanh, nên kẻ tấn công có thể thử hàng tỷ mật khẩu mỗi giây trên một bảng dữ liệu bị rò rỉ. Dùng bcrypt hoặc argon2, vốn được thiết kế để chậm (100–300ms mỗi lần băm) và có salt riêng cho từng mật khẩu.

security.py
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["argon2"], deprecated="auto")

def hash_password(raw: str) -> str:
    return pwd_context.hash(raw)

def verify_password(raw: str, hashed: str) -> bool:
    return pwd_context.verify(raw, hashed)

# Lỗi tinh tế cần tránh: trả lỗi khác nhau cho "email không tồn tại"
# và "sai mật khẩu" → cho phép kẻ tấn công dò xem email nào đã đăng ký.
# Luôn trả CÙNG MỘT thông báo lỗi chung cho cả hai trường hợp.

Rate limiting: token bucket

Giới hạn tần suất bảo vệ chi phí và chống lạm dụng. Thuật toán phổ biến nhất là token bucket: mỗi định danh (IP, user, API key) có một "xô" chứa B token, nạp lại với tốc độ r token/giây; mỗi request tiêu một token; xô cạn thì trả về 429 kèm header Retry-After.

Serving LLM & RAG

LLM là một hệ thống xác suất — coi nó như một nguồn dữ liệu mạnh nhưng không đáng tin, và bọc nó trong cùng hàng rào validate bạn dùng cho input của người dùng.

Vì sao cần một LLM Gateway riêng

Thay vì để mọi service trong hệ thống gọi thẳng API của OpenAI/Anthropic, hãy bọc nó trong một service FastAPI trung gian. Một nơi để quản lý: xác thực, prompt template, guardrail, cache, đo chi phí theo từng user, và chuyển đổi nhà cung cấp mà không cần sửa code phía client.

Stream token theo thời gian thực bằng SSE

Trải nghiệm chat chỉ "mượt" khi token đầu tiên xuất hiện sau vài trăm mili-giây, thay vì người dùng phải đợi toàn bộ câu trả lời (có thể mất chục giây). StreamingResponse bọc một async generator, đẩy từng đoạn dữ liệu ngay khi sinh ra.

routers/chat.py — stream token qua SSE
from fastapi.responses import StreamingResponse
from pydantic import BaseModel, Field

class ChatIn(BaseModel):
    message: str = Field(min_length=1, max_length=4000)
    max_tokens: int = Field(default=512, le=2048)  # trần chi phí, không chỉ trần kỹ thuật

@router.post("/chat/stream")
async def chat_stream(req: ChatIn, user: User = Depends(get_current_user)):
    await guardrails.check_input(req.message)  # kiểm tra TRƯỚC khi gọi model

    async def token_stream():
        usage = {"out_tokens": 0}
        try:
            async for chunk in llm_client.stream(req.message, max_tokens=req.max_tokens):
                usage["out_tokens"] += 1
                yield f"data: {chunk.json()}\n\n"
            yield "data: [DONE]\n\n"
        finally:
            # chạy NGAY CẢ KHI client ngắt kết nối giữa chừng — đo chi phí đúng
            await metering.record(user.id, usage)

    return StreamingResponse(token_stream(), media_type="text/event-stream")
Đừng quên khối finally

Client ngắt kết nối giữa chừng (đóng tab, mất mạng) là chuyện xảy ra liên tục với streaming. Nếu việc đo token tiêu thụ nằm ngoài finally, bạn sẽ mất dấu chi phí của chính những request bị huỷ giữa chừng — thường lại là những request tốn kém nhất.

Kiến trúc RAG: từ câu hỏi đến câu trả lời có trích dẫn

Retrieval-Augmented Generation gồm hai luồng riêng biệt: nạp dữ liệu (chia nhỏ tài liệu → tạo embedding → lưu vào vector DB) chạy một lần hoặc định kỳ, và truy vấn (embed câu hỏi → tìm đoạn liên quan → đưa vào prompt → sinh câu trả lời) chạy mỗi lần người dùng hỏi.

Hình 6 — Kiến trúc RAG: nạp dữ liệu chạy nền, truy vấn chạy theo từng câu hỏi, ACL nằm ở tầng database.

Validate đầu ra của LLM như validate input không tin cậy

Một mô hình ngôn ngữ là nguồn dữ liệu xác suất — coi đầu ra của nó giống hệt như input chưa qua kiểm chứng từ người dùng. Ép nó trả JSON, parse qua Pydantic, và nếu lỗi validate thì gửi lại lỗi cho model tự sửa một lần trước khi bỏ cuộc.

services/extract.py — ép LLM tuân theo schema
from pydantic import BaseModel, ValidationError, Field
from decimal import Decimal
from typing import Literal

class HoaDonTrichXuat(BaseModel):
    nha_cung_cap: str
    tong_tien: Decimal = Field(ge=0)
    don_vi: Literal["VND", "USD"]

async def trich_xuat_hoa_don(van_ban: str) -> HoaDonTrichXuat:
    prompt = PROMPT_MAU.format(text=van_ban)
    for lan_thu in range(2):
        raw = await llm.complete(prompt, json_mode=True)
        try:
            return HoaDonTrichXuat.model_validate_json(raw)
        except ValidationError as e:
            # Đưa lỗi validate ngược lại cho model để nó tự sửa — thường thành công ngay lần 2
            prompt = f"{prompt}\n\nKết quả trước bị lỗi:\n{e}\nHãy trả JSON đã sửa."
    raise HTTPException(502, "Mô hình không trả về dữ liệu hợp lệ")

Prompt injection: phòng thủ phân lớp

Không có cách nào chặn prompt injection 100%, nhưng các lớp phòng thủ kết hợp giúp giảm rủi ro đáng kể: cấp quyền tối thiểu cho mọi tool mà LLM có thể gọi, không bao giờ để đầu ra của model tự động thực thi hành động nhạy cảm mà không qua xác nhận, và luôn validate output trước khi trả về hoặc dùng nó để gọi hành động khác.

  • Mọi response từ LLM đều bọc model_version để biết model nào đã sinh ra nó
  • max_tokens là một ràng buộc Pydantic — đồng thời là giới hạn chi phí
  • ACL của RAG nằm trong câu SQL, không nằm trong lời dặn prompt
  • Câu trả lời không có trích dẫn hợp lệ → từ chối trả lời thay vì bịa
  • finally luôn chạy để đo token dùng được, kể cả khi client ngắt kết nối

Production Checklist

Danh sách kiểm tra cuối cùng trước khi đưa service ra phục vụ người dùng thật — tổng hợp lại các nguyên tắc đã nêu ở những chương trước.

Liveness vs Readiness — đừng nhầm lẫn hai loại probe

Liveness trả lời "tiến trình này có bị treo không" — nếu sai, Kubernetes sẽ restart pod. Readiness trả lời "pod này có sẵn sàng nhận traffic không" — nếu sai, pod chỉ bị rút khỏi load balancer, không bị restart. Nhầm hai loại này là một lỗi kinh điển: nếu liveness probe lại đi kiểm tra database, một lần database chập chờn 30 giây sẽ khiến toàn bộ pod bị restart đồng loạt — biến một sự cố nhỏ thành downtime toàn diện.

routers/health.py
@app.get("/health/live")
def live():
    """Chỉ hỏi: tiến trình có phản hồi không. KHÔNG đụng tới dependency."""
    return {"status": "ok"}

@app.get("/health/ready")
async def ready(db: Session = Depends(get_db)):
    """Hỏi: có sẵn sàng nhận traffic không. Kiểm tra dependency thiết yếu."""
    await db.execute(text("SELECT 1"))
    if not app.state.model_loaded:
        raise HTTPException(503, "Mô hình chưa nạp xong")
    return {"status": "ready"}

Cảnh báo theo triệu chứng, không theo nguyên nhân

Báo động khi "CPU vượt 80%" là một anti-pattern — CPU cao có thể hoàn toàn lành mạnh (đang tận dụng tốt tài nguyên). Hãy báo động theo SLO bị vi phạm (tỉ lệ lỗi tăng, độ trễ p99 vượt ngưỡng) — những gì người dùng thực sự cảm nhận được. Nguyên nhân (CPU, pool, queue depth) chỉ nên xuất hiện trên dashboard bạn mở ra sau khi đã được báo động.

Checklist trước khi go-live

  • Mọi endpoint đều có response_model whitelist tường minh field trả về
  • Connection pool được tính toán theo Little's Law, không đoán mò
  • Mọi lệnh gọi ra ngoài (DB, API, LLM) đều có timeout ngắn hơn timeout của client gọi nó
  • Liveness probe không đụng tới database; readiness probe có đụng
  • Secrets nằm trong secret manager, không nằm trong code hay image
  • Đã chạy load test tới điểm "knee" của đường cong độ trễ, không chỉ test một request đơn lẻ
  • Có runbook cho 3-5 kịch bản sự cố thường gặp nhất (DB down, LLM provider down, pool cạn)
  • Đã thử rollback ít nhất một lần trên thực tế, không chỉ trên giấy
Tóm gọn cả tài liệu trong một câu

Một API FastAPI sống sót được trong production không phải nhờ framework, mà nhờ kỷ luật ở bốn ranh giới: ranh giới giữa code chặn và code bất đồng bộ, ranh giới giữa dữ liệu tin cậy và chưa tin cậy (kể cả khi nguồn đó là một LLM), ranh giới giữa "đã đăng nhập" và "được phép", và ranh giới giữa tài nguyên hữu hạn (pool, token, tiền) và lưu lượng truy cập vô hạn.
