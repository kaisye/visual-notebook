Git là máy ảnh, GitHub là ngôi nhà chung

Một buổi tối, bạn sửa xong tính năng đăng nhập. Sáng hôm sau, mọi thứ lỗi tung tóe. May mắn là Git đã “chụp ảnh” từng phiên bản code.

🌿 Git

Công cụ quản lý phiên bản: ghi lại lịch sử thay đổi, so sánh code, quay lại bản cũ và làm việc song song bằng branch.

🏠 GitHub

Nền tảng cloud cho repository: lưu code, giao việc bằng Issues, review bằng Pull Request, quản lý sprint và chạy tự động hóa.

Hiểu nhanh: Git giữ lịch sử, GitHub giúp cả đội phối hợp.

Issues: biến “ý tưởng mơ hồ” thành việc làm được

Trong nhóm, câu “màn hình login bị lỗi” là chưa đủ. Một Issue tốt giúp cả đội biết lỗi ở đâu, cần sửa gì và thế nào là xong.

Bug

Ghi bước tái hiện, môi trường, ảnh chụp và kết quả mong đợi.

Task

Mô tả việc cần làm, người phụ trách, deadline và tiêu chí hoàn thành.

Ý tưởng

Lưu đề xuất cải tiến để thảo luận, ưu tiên và đưa vào sprint phù hợp.

Công thức Issue gọn

Bối cảnh → Việc cần làm → Tiêu chí hoàn thành → Tài liệu/ảnh liên quan.

🤝

Pull Request: cánh cửa kiểm duyệt trước khi vào main

Bạn không bước thẳng vào phòng điều khiển và bấm nút deploy. Bạn mở Pull Request để cả đội cùng soi đèn: logic ổn chưa, test qua chưa, có ảnh hưởng ai không?

PR nên có

  • Mô tả thay đổi.
  • Link Issue liên quan.
  • Ảnh demo hoặc video ngắn nếu có UI.
  • Checklist đã test.

Review nên nhìn

  • Logic nghiệp vụ.
  • Bảo mật và dữ liệu nhạy cảm.
  • Hiệu năng.
  • Dễ đọc, dễ bảo trì.
  • Test đủ tình huống.

PR không phải “bắt lỗi nhau”, mà là bảo hiểm chất lượng cho cả đội.

👥

Luồng làm việc nhóm: từ task đến merge

Khi quy trình rõ, nhóm không còn hỏi “ai đang làm gì?” mỗi 15 phút. Mỗi task tự kể câu chuyện của nó trên GitHub.

Hành trình đề xuất

  1. Nhận Issue và làm rõ yêu cầu.
  2. Tạo branch riêng từ main.
  3. Code, commit nhỏ theo từng phần.
  4. Mở Pull Request và gắn reviewer.
  5. Sửa theo góp ý, đợi CI pass.
  6. Merge khi đã approve và không conflict.

Quy ước giúp đội chạy mượt

  • feature/payment-ui cho tính năng mới.
  • fix/login-error cho sửa lỗi.
  • feat:, fix:, docs: cho commit message.
  • Không push trực tiếp vào main.

CI/CD: robot kiểm tra trước khi code ra sân khấu

Trước khi bản diễn ra mắt, robot hậu trường sẽ build, lint, test. Nếu có lỗi, nó giơ biển đỏ ngay trong Pull Request.

Lint

Build/Deploy

CI

Tự động tích hợp: build, lint, test mỗi khi có code mới để phát hiện lỗi sớm.

CD

Tự động chuẩn bị hoặc triển khai lên staging/production khi code đạt chuẩn.

🧪

Workflow mẫu: bài kiểm tra tự động cho Pull Request

Một file nhỏ trong thư mục .github/workflows có thể biến GitHub thành người gác cổng chất lượng.

name: CI on: pull_request: branches: [main] push: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: 20 - run: npm ci - run: npm run lint - run: npm test - run: npm run build
Ý nghĩa: chỉ nên merge khi các bước kiểm tra chạy xanh. Nếu đỏ, sửa code trước khi gộp.
🚀

Tóm tắt: làm GitHub như một đội chuyên nghiệp

Khi mọi người cùng theo một nhịp, GitHub không còn là “chỗ cất code” mà trở thành dây chuyền sản xuất phần mềm.

✓
Chia việc rõ: dùng Issues, Labels, Milestones và Projects.
✓
Làm việc an toàn: code trên branch riêng, không push thẳng vào main.
✓
Review kỹ: Pull Request có mô tả, checklist test và người duyệt.
✓
Tự động hóa: GitHub Actions chạy lint, test, build và deploy.
✓
Hành động ngay: tạo một repo thử nghiệm, mở PR đầu tiên và bật CI cơ bản.

Commit nhỏ, review kỹ, deploy tự tin! 🎉
